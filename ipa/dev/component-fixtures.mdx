# Component fixtures

This page is not shipped — `draft: true` excludes it from production. It opens
with a full, production-shaped guideline, then isolates each component in its
own section.

## Full example

A complete guideline as it renders in production: guidance prose with an
admonition, a correct and an incorrect worked example, and a multi-step
evaluation workflow. The examples + workflow are wrapped in a
`<Guideline.Details>` so they collapse into a single **"Details"** accordion
below the prose, and the `Informational`/`Enforced` tag no longer renders in the
header — the `informational` field stays in the metadata.

<Guidelines>

<Guideline id="IPA-000-must-use-american-english" given="spec" enforcement="rule">

API producers **must** use American English across the API — resource names,
field names, descriptions, and enum values.

:::note

The MongoDB Style Guide follows Merriam-Webster's Collegiate Dictionary.

:::

<Guideline.Details>

<Example.Correct>

```yaml
enum: [CANCELED, COMPLETED]
```

<Example.Reason>
  "CANCELED" uses the American English spelling (one "l").
</Example.Reason>

</Example.Correct>

<Example.Incorrect>

```yaml
enum: [CANCELLED, COMPLETED]
```

<Example.Reason>
  "CANCELLED" is British English and inconsistent with other MongoDB APIs.
</Example.Reason>

</Example.Incorrect>

<Workflow>
  <Workflow.Step>
    Scan every user-facing string in the specification (descriptions, summaries,
    enum values, field names).
  </Workflow.Step>
  <Workflow.Step>
    Check each for British English spellings (e.g. "cancelled", "colour",
    "authorise").
  </Workflow.Step>
  <Workflow.Step>Flag any non-American spelling as a violation.</Workflow.Step>
</Workflow>

</Guideline.Details>

</Guideline>

</Guidelines>

## `<Guidelines>` and `<Guideline>`

`<Guidelines>` is an ordered-list wrapper that numbers its `<Guideline>`
children and draws separators between them. A `<Guideline>` resolves its own
`state` and renders its header, body, and footer.

### Numbered list with separators

<Guidelines>
  <Guideline
    id="IPA-000-must-render-first-guideline"
    enforcement="rule"
    state="adopt"
    effort="check"
    given="operation"
  >
    First guideline appears at the top of the card.
  </Guideline>
  <Guideline
    id="IPA-000-should-render-second-guideline"
    enforcement="automatable"
    implementation
    state="adopt"
    effort="reason"
    given="operation"
  >
    Second guideline is separated from the first by a top border.
  </Guideline>
  <Guideline
    id="IPA-000-may-render-third-guideline"
    enforcement="advisory"
    state="adopt"
    effort="explore"
  >
    Third guideline shows that separator styling applies between every adjacent
    pair.
  </Guideline>
</Guidelines>

### Standalone (no numbering)

A `<Guideline>` rendered outside `<Guidelines>` shows no numbering circle and
renders as a `<div>` rather than an `<li>`.

<Guideline
  id="IPA-000-must-render-standalone"
  enforcement="advisory"
  state="adopt"
  effort="check"
>
  Without a `<Guidelines>` wrapper, no number badge is shown and no list
  semantics are applied.
</Guideline>

</Example.Correct>

```yaml
{}
```

</Example.Incorrect>

## `<Example.Reason>`

`<Example.Reason>` is the short prose explanation inside an example — why the
pattern is correct or incorrect. It renders beneath the code block under the
example's tinted hairline, with an inline "Why:" lead-in in the verdict color.

### Inside a correct example

<Example.Correct>

```yaml
PATCH /clusters/{clusterId}
Content-Type: application/merge-patch+json
```

<Example.Reason>
  Merge-patch is simpler for clients — send only the fields to change.
</Example.Reason>

</Example.Correct>

### Inside an incorrect example

<Example.Incorrect>

```yaml
components:
  schemas:
    Cluster:
      properties:
        _id: { type: string }
        __v: { type: integer }
```

<Example.Reason>
  Exposing `_id` and `__v` leaks MongoDB document internals into the API
  surface, coupling clients to the storage layer.
</Example.Reason>

</Example.Incorrect>

## `<Workflow>`

`<Workflow>` renders the manual evaluation steps of a guideline as a numbered
reading list inside a collapsible accordion.

:::note

The site sets `showWorkflows: true` in `docusaurus.config.ts`. Set it to `false`
to hide workflows everywhere.

:::

### Inside a guideline (lone workflow, not wrapped)

A guideline with a workflow but no examples keeps the workflow as its own
accordion — it is not wrapped in a redundant outer "Details" accordion.

## Grouping behavior

To collapse a guideline's supplementary content, wrap it in
`<Guideline.Details>` (see [Full example](#full-example) above). Whatever you
put inside — examples, workflow, notes — renders in one collapsed "Details"
accordion below the prose; pass `title="…"` to relabel it. Leave content outside
the wrapper to keep it inline (e.g. a lone `<Workflow>` that should render as
its own accordion).
